## Огляд архітектури

API PayPaste складається з трьох типів endpoint'ів:

| Тип             | Автентифікація                             | Опис                                                                     |
| --------------- | ------------------------------------------ | ------------------------------------------------------------------------ |
| **REST API**    | JWT Bearer token (платний: 1 кредит/запит) | Програмний доступ до паст (CRUD)                                         |
| **Сесійні API** | PHP Session (`$_SESSION`)                  | Взаємодія з браузера (OAuth, Passkey, завантаження, перевірка замовлень) |
| **Webhooks**    | Токен/підпис провайдера                    | Колбеки від зовнішніх сервісів (платежі, реклама)                        |

### Структура файлів

```
api/
├── api_helper.php          # Хелпери: authenticate_api(), json_response()
├── auth_token.php          # Обмін API-ключа на JWT
├── check_order_status.php  # Перевірка статусу замовлення на поповнення
├── download.php            # Проксі-завантаження файлів з перевіркою доступу
├── oauth.php               # GitHub та Telegram OAuth
├── passkey.php             # WebAuthn/FIDO2 Passkey API
├── pastes.php              # REST API для паст (GET/POST/DELETE)
└── webhooks/
    ├── donatello.php       # Webhook від Donatello (fiat-платежі)
    ├── telegram_stars.php  # Webhook від Telegram Stars (XTR-платежі)
    └── verify_ad.php       # AJAX-підтвердження перегляду реклами
```

## Автентифікація API

REST API використовує **двоступеневу автентифікацію**:

1. Користувач має **API-ключ** (генерується в налаштуваннях, зберігається у БД `users.api_key`)
2. Обмінює API-ключ на **JWT-токен** через `auth_token.php`
3. Надсилає JWT-токен у заголовку `Authorization: Bearer <token>` з кожним запитом

### Отримання JWT-токена (`auth_token.php`)

Обмінює довгостроковий API-ключ на короткостроковий JWT-токен.

| Параметр         | Значення                                                   |
| ---------------- | ---------------------------------------------------------- |
| **URL**          | `/api/auth_token.php`                                      |
| **Метод**        | `POST`                                                     |
| **Content-Type** | `application/json` або `application/x-www-form-urlencoded` |

#### Параметри запиту

| Поле      | Тип    | Обов'язковий | Опис                               |
| --------- | ------ | ------------ | ---------------------------------- |
| `api_key` | string | ✅           | API-ключ користувача з налаштувань |

#### Приклад запиту

```bash
# JSON
curl -X POST https://example.com/api/auth_token.php \
  -H "Content-Type: application/json" \
  -d '{"api_key": "ak_your_api_key_here"}'

# POST-форма
curl -X POST https://example.com/api/auth_token.php \
  -d "api_key=ak_your_api_key_here"
```

#### Відповідь при успіху (200)

```json
{
  "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "token_type": "Bearer",
  "expires_in": 604800
}
```

| Поле           | Опис                                        |
| -------------- | ------------------------------------------- |
| `access_token` | JWT-токен (алгоритм HS256)                  |
| `token_type`   | Завжди `"Bearer"`                           |
| `expires_in`   | Час дії токена в секундах (604800 = 7 днів) |

#### JWT Payload структура

```json
{
  "sub": 42,
  "nickname": "user123",
  "iat": 1746300000,
  "exp": 1746904800
}
```

| Поле       | Опис                                              |
| ---------- | ------------------------------------------------- |
| `sub`      | ID користувача                                    |
| `nickname` | Нікнейм користувача                               |
| `iat`      | Час видачі (UNIX timestamp)                       |
| `exp`      | Час закінчення дії (UNIX timestamp, iat + 7 днів) |

#### Коди помилок

| Код | Умова                                                                  |
| --- | ---------------------------------------------------------------------- |
| 405 | Метод не POST                                                          |
| 400 | Порожній `api_key`                                                     |
| 401 | Невірний API-ключ                                                      |
| 429 | Занадто багато спроб (Rate Limit: 5 спроб/60с за ключем, 150/хв за IP) |

> ⚠️ **Rate Limiting**: Обмеження 5 запитів на 60 секунд за API-ключем + м'яке IP-обмеження 150/хв.

### Механізм автентифікації запитів (`api_helper.php`)

Функція `authenticate_api()` викликається на початку кожного REST-endpoint'у.

#### Алгоритм автентифікації

```
1. Читаємо заголовок Authorization
2. Перевіряємо формат "Bearer <token>"
3. Декодуємо JWT → отримуємо payload
4. Перевіряємо термін дії (exp)
5. Знаходимо користувача за payload.sub
6. Rate Limit: 60 запитів/хв на користувача
7. Перевіряємо баланс (мін. 1 кредит)
8. Списуємо 1 кредит (тип транзакції: api_usage)
9. Повертаємо об'єкт User
```

> 💰 **Платність**: Кожен REST API-запит списує **1 кредит** з балансу користувача (transaction type: `api_usage`). Якщо баланс < 1, повертається помилка 402.

#### Коди помилок автентифікації

| Код | Умова                                         |
| --- | --------------------------------------------- |
| 401 | Відсутній/некоректний заголовок Authorization |
| 401 | Недійсний або протермінований JWT             |
| 401 | Користувача не знайдено в БД                  |
| 429 | Перевищено Rate Limit (60 запитів/хв)         |
| 402 | Недостатньо кредитів (баланс < 1)             |

#### Хелпер `json_response()`

Функція для відправки JSON-відповідей:

```php
json_response($data, $status = 200);
```

Встановлює `Content-Type: application/json; charset=utf-8`, код статусу та виводить JSON з `JSON_UNESCAPED_UNICODE | JSON_PRETTY_PRINT`.

## REST API: Пасти (`pastes.php`)

Повний CRUD для паст через REST API. Усі запити вимагають автентифікацію через JWT Bearer token (див. [вище](#автентифікація-api)).

| Метод    | URL                             | Опис            |
| -------- | ------------------------------- | --------------- |
| `GET`    | `/api/pastes.php?id=<paste_id>` | Отримання пасти |
| `POST`   | `/api/pastes.php`               | Створення пасти |
| `DELETE` | `/api/pastes.php?id=<paste_id>` | Видалення пасти |

> Також підтримується PATH_INFO: `/api/pastes.php/<paste_id>` як альтернатива параметру `?id=`.

---

### GET — Отримання пасти

Отримує вміст пасти за ID. Якщо паста платна і користувач ще не розблокував її — **автоматично списуються кредити** і вона розблоковується.

#### Параметри запиту

| Поле | Розташування        | Тип    | Обов'язковий | Опис                           |
| ---- | ------------------- | ------ | ------------ | ------------------------------ |
| `id` | Query або PATH_INFO | string | ✅           | ID пасти (напр. `p_abc123def`) |

#### Приклад запиту

```bash
curl https://example.com/api/pastes.php?id=p_abc123def \
  -H "Authorization: Bearer eyJhbGci..."
```

#### Відповідь при успіху (200)

```json
{
  "id": "p_abc123def",
  "title": "Мій код",
  "content": "print('hello')",
  "author": 42,
  "created_at": "2026-05-10 14:30:00",
  "expires_at": "2026-05-17 14:30:00",
  "is_paid": true,
  "view_cost": 50,
  "language": "python",
  "tags": ["backend", "api"]
}
```

| Поле         | Тип          | Опис                          |
| ------------ | ------------ | ----------------------------- |
| `id`         | string       | Унікальний ID пасти           |
| `title`      | string       | Назва пасти                   |
| `content`    | string       | Вміст пасти                   |
| `author`     | int          | ID автора                     |
| `created_at` | string       | Дата створення (Y-m-d H:i:s)  |
| `expires_at` | string\|null | Дата закінчення TTL           |
| `is_paid`    | bool         | Чи є паста платною            |
| `view_cost`  | int          | Вартість перегляду (кредитів) |
| `language`   | string       | Мова підсвітки синтаксису     |
| `tags`       | string[]     | Масив тегів                   |

#### Особливості поведінки

- **Приватна паста**: Доступна лише автору та адміністраторам (інакше 403)
- **Платна паста (не розблокована)**: Автоматично списується `view_cost` кредитів → автор отримує кредити (sale) → паста розблоковується. Якщо баланс недостатній — помилка 402
- **Прострочена паста**: Повертається 404 (видаляється через лінивий TTL-механізм)

#### Коди помилок

| Код | Умова                                                |
| --- | ---------------------------------------------------- |
| 400 | Не вказано `id`                                      |
| 403 | Приватна паста, доступ заборонено                    |
| 402 | Платна паста, недостатньо кредитів для розблокування |
| 404 | Пасту не знайдено або термін дії закінчився          |

### POST — Створення пасти

Створює нову пасту. За замовчуванням паста створюється зі статусом модерації `pending` і ставиться у чергу на перевірку.

#### Параметри запиту (JSON body)

| Поле          | Тип    | Обов'язковий | За замовчуванням    | Опис                          |
| ------------- | ------ | ------------ | ------------------- | ----------------------------- |
| `title`       | string | ❌           | `"Без назви (API)"` | Назва пасти                   |
| `content`     | string | ❌           | `""`                | Вміст пасти                   |
| `is_private`  | bool   | ❌           | `false`             | Приватна видимість            |
| `is_paid`     | bool   | ❌           | `false`             | Платна паста                  |
| `view_cost`   | int    | ❌           | `0`                 | Вартість перегляду (кредитів) |
| `ttl_minutes` | int    | ❌           | `10080` (7 днів)    | Час життя у хвилинах          |
| `language`    | string | ❌           | `"plaintext"`       | Мова підсвітки                |
| `tags`        | string | ❌           | `""`                | Теги (розділені комами)       |

#### Приклад запиту

```bash
curl -X POST https://example.com/api/pastes.php \
  -H "Authorization: Bearer eyJhbGci..." \
  -H "Content-Type: application/json" \
  -d '{
    "title": "Приклад API",
    "content": "console.log(\"hello\");",
    "is_paid": true,
    "view_cost": 25,
    "language": "javascript",
    "tags": "demo,api"
  }'
```

#### Відповідь при успіху (201)

```json
{
  "id": "p_new123abc",
  "url": "https://example.com/view.php?id=p_new123abc",
  "title": "Приклад API",
  "moderation_status": "pending",
  "creation_fee": 3,
  "remaining_credits": 96
}
```

| Поле                | Тип    | Опис                         |
| ------------------- | ------ | ---------------------------- |
| `id`                | string | ID створеної пасти           |
| `url`               | string | URL для перегляду            |
| `title`             | string | Назва                        |
| `moderation_status` | string | Статус модерації (`pending`) |
| `creation_fee`      | int    | Списані кредити за створення |
| `remaining_credits` | int    | Залишок балансу користувача  |

#### Розрахунок creation_fee

Платна паста (`is_paid=true`) списує `ceil(strlen(content) / 10)` кредитів як плату за створення. Безкоштовні пасти не мають плати за створення.

#### Модерація

1. **Локальна перевірка** (`Moderation::localCheck()`) — виконується синхронно перед створенням. Якщо знайдено порушення — паста не створюється (400)
2. **Зовнішня модерація** — паста створюється зі статусом `pending` і ставиться у чергу (`Queue::push(TYPE_MODERATION_CHECK, ...)`)

#### Коди помилок

| Код | Умова                                               |
| --- | --------------------------------------------------- |
| 400 | Контент не пройшов локальну перевірку модерації     |
| 400 | Інша помилка валідації (напр. порожній контент)     |
| 402 | Недостатньо кредитів для creation_fee платної пасти |
| 405 | Метод не POST                                       |

### DELETE — Видалення пасти

Видаляє пасту. Доступно лише автору пасти.

#### Параметри запиту

| Поле | Розташування        | Тип    | Обов'язковий | Опис     |
| ---- | ------------------- | ------ | ------------ | -------- |
| `id` | Query або PATH_INFO | string | ✅           | ID пасти |

#### Приклад запиту

```bash
curl -X DELETE "https://example.com/api/pastes.php?id=p_abc123def" \
  -H "Authorization: Bearer eyJhbGci..."
```

#### Відповідь при успіху (200)

```json
{
  "success": true,
  "message": "Пасту видалено."
}
```

#### Коди помилок

| Код | Умова                                        |
| --- | -------------------------------------------- |
| 400 | Не вказано `id`                              |
| 403 | Паста належить іншому користувачу (не автор) |
| 404 | Пасту не знайдено                            |
| 405 | Метод не GET/POST/DELETE                     |

## Сесійні API

Ці endpoint'и не використовують JWT, а покладаються на **PHP-сесію** (`$_SESSION`). Викликаються з браузера (JavaScript fetch або перенаправлення).

---

### Перевірка статусу замовлення (`check_order_status.php`)

Дозволяє клієнту перевірити, чи було завершено поповнення балансу. Використовується для polling-перевірки під час очікування оплати.

| Параметр           | Значення                      |
| ------------------ | ----------------------------- |
| **URL**            | `/api/check_order_status.php` |
| **Метод**          | `GET`                         |
| **Автентифікація** | Сесія (не обов'язкова)        |

#### Параметри запиту

| Поле       | Розташування | Тип    | Обов'язковий | Опис                                 |
| ---------- | ------------ | ------ | ------------ | ------------------------------------ |
| `order_id` | Query        | string | ✅           | ID замовлення (напр. `order_abc123`) |

#### Відповідь — замовлення знайдено (200)

```json
{
  "exists": true,
  "status": "completed",
  "amount_credits": 500,
  "current_balance": 600,
  "service": "donatello"
}
```

| Поле              | Тип    | Опис                                       |
| ----------------- | ------ | ------------------------------------------ |
| `exists`          | bool   | Чи існує замовлення                        |
| `status`          | string | Статус: `pending`, `completed`             |
| `amount_credits`  | int    | Нараховані кредити                         |
| `current_balance` | int    | Поточний баланс користувача                |
| `service`         | string | Платіжний сервіс (`donatello`, `tg_stars`) |

#### Відповідь — замовлення не знайдено

```json
{
  "exists": false
}
```

#### Коди помилок

| Код | Умова                 |
| --- | --------------------- |
| 400 | Не вказано `order_id` |

> 💡 **Особливість**: Після завершення замовлення викликається `session_write_close()` для примусового збереження сесії, щоб наступний запит користувача вже бачив оновлений баланс.

### Завантаження файлів (`download.php`)

Проксі-скрипт для безпечного завантаження файлів прикріплених до паст. Файли зберігаються в `data/uploads/` (захищено `.htaccess` від прямого доступу), цей скрипт перевіряє права доступу перед віддачею.

| Параметр           | Значення                           |
| ------------------ | ---------------------------------- |
| **URL**            | `/api/download.php`                |
| **Метод**          | `GET`                              |
| **Автентифікація** | Сесія (для платних/приватних паст) |

#### Параметри запиту

| Поле | Розташування | Тип    | Обов'язковий | Опис     |
| ---- | ------------ | ------ | ------------ | -------- |
| `id` | Query        | string | ✅           | ID пасти |

#### Логіка перевірки доступу

```
1. Публічна + безкоштовна → доступ відкритий
2. Авторизований + автор пасти → доступ відкритий
3. Авторизований + розблокував пасту → доступ відкритий
4. Інакше → 403 Forbidden
```

#### Поведінка віддачі файлу

| Тип файлу                        | Content-Disposition | MIME                                            |
| -------------------------------- | ------------------- | ----------------------------------------------- |
| Зображення (jpg, jpeg, png, gif) | `inline`            | `image/*`                                       |
| Інші (pdf, zip, txt, ...)        | `attachment`        | Відповідний MIME або `application/octet-stream` |

Пошук файлу: `glob(data/uploads/<paste_id>.*)` — файл має мати назву `paste_id.<ext>`.

#### Коди помилок

| Код | Умова                                                        |
| --- | ------------------------------------------------------------ |
| 400 | Не вказано `id`                                              |
| 403 | Немає доступу (не автор, не розблоковано)                    |
| 404 | Пасту не знайдено / термін дії закінчився / файл не знайдено |

### OAuth авторизація (`oauth.php`)

Обробляє авторизацію через **GitHub** та **Telegram**. Підтримує три сценарії: вхід, реєстрація, прив'язка до існуючого акаунта, та підтвердження видалення акаунта.

| Параметр           | Значення                                     |
| ------------------ | -------------------------------------------- |
| **URL**            | `/api/oauth.php?provider=<provider>`         |
| **Метод**          | `GET` (перенаправлення від OAuth-провайдера) |
| **Автентифікація** | Сесія (для прив'язки/видалення)              |

#### GitHub OAuth Flow

```
Крок 1: Клієнт → /api/oauth.php?provider=github
         ↪ Перенаправлення на github.com/login/oauth/authorize
            Параметри: client_id, redirect_uri, scope=read:user user:email

Крок 2: GitHub callback → /api/oauth.php?provider=github&code=<code>
         ↪ Обмін code на access_token (POST github.com/login/oauth/access_token)
         ↪ Отримання профілю (GET api.github.com/user)
         ↪ Якщо email відсутній — отримання з api.github.com/user/emails
         ↪ Авторизація/реєстрація через AuthService::oauthLogin()
```

#### Telegram Login Widget Flow

```
Крок 1: Telegram Widget → /api/oauth.php?provider=telegram&id=...&hash=...
         ↪ Перевірка HMAC-SHA256 підпису
            Формула: hash=HMAC-SHA256(sorted_data, SHA256(bot_token))
         ↪ Авторизація/реєстрація через AuthService::oauthLogin()
```

**Алгоритм перевірки Telegram підпису:**

1. Видалити `hash`, `provider`, `confirm_delete_oauth` з `$_GET`
2. Відсортувати ключі за алфавітом
3. Склеїти у `key=value\nkey=value`
4. Обчислити `secret_key = SHA256(bot_token, raw_output=true)`
5. Обчислити `hash = HMAC-SHA256(data_check_string, secret_key)`
6. Порівняти з `$_GET['hash']` через `hash_equals()`

#### Сценарії обробки

| Сценарій                    | Умова                                         | Дія                                                                 |
| --------------------------- | --------------------------------------------- | ------------------------------------------------------------------- |
| **Вхід/реєстрація**         | `$_SESSION['user_id']` не встановлено         | `AuthService::oauthLogin()` → автоматична реєстрація з 100 кредитів |
| **Прив'язка**               | `$_SESSION['user_id']` встановлено            | `AuthService::linkOAuth()` → прив'язка провайдера до акаунта        |
| **Підтвердження видалення** | `confirm_delete_oauth` у GET + співпадіння ID | `$_SESSION['oauth_confirmed_delete'] = true`                        |

#### Параметри запиту

| Поле                   | Тип    | Обов'язковий | Опис                            |
| ---------------------- | ------ | ------------ | ------------------------------- |
| `provider`             | string | ✅           | `github` або `telegram`         |
| `code`                 | string | Для GitHub   | Код авторизації від GitHub      |
| `hash`                 | string | Для Telegram | HMAC-підпис від Telegram Widget |
| `id`                   | int    | Для Telegram | Telegram ID користувача         |
| `confirm_delete_oauth` | flag   | ❌           | Підтвердження видалення акаунта |

#### Редиректи після обробки

| Сценарій                | Редирект                               |
| ----------------------- | -------------------------------------- |
| Вхід/реєстрація         | `../index.php`                         |
| Прив'язка/підтвердження | `../settings.php`                      |
| Помилка                 | Відображається `die()` з повідомленням |

> 📝 **Примітка**: Для Telegram-користувачів створюється фіктивний email `telegram_{id}@paypaste.local`. Нові користувачі через OAuth автоматично отримують 100 кредитів.

### WebAuthn / Passkeys (`passkey.php`)

API для реєстрації, входу та управління Passkeys (WebAuthn/FIDO2). Усі дії доступні через параметр `action` у query string.

| Параметр           | Значення                                                               |
| ------------------ | ---------------------------------------------------------------------- |
| **URL**            | `/api/passkey.php?action=<action>`                                     |
| **Автентифікація** | Сесія (обов'язкова для register_finish, delete, confirm_delete_finish) |

#### Доступні дії (action)

| Action                  | Метод | Опис                                          | Автентифікація |
| ----------------------- | ----- | --------------------------------------------- | -------------- |
| `register_start`        | GET   | Початок реєстрації: генерація челенджу        | Не обов'язкова |
| `register_finish`       | POST  | Завершення реєстрації: верифікація відповіді  | Сесія + CSRF   |
| `login_start`           | GET   | Початок входу: генерація челенджу             | Немає          |
| `login_finish`          | POST  | Завершення входу: верифікація підпису         | Немає          |
| `confirm_delete_finish` | POST  | Підтвердження видалення акаунта через Passkey | Сесія          |
| `delete`                | POST  | Видалення Passkey зі списку                   | Сесія + CSRF   |

#### `register_start` — Початок реєстрації

Генерує WebAuthn-челендж для створення нових облікових даних на пристрої.

**Параметри:**

| Поле       | Розташування | Тип    | За замовчуванням | Опис                   |
| ---------- | ------------ | ------ | ---------------- | ---------------------- |
| `nickname` | Query        | string | `"PasskeyUser"`  | Назва для нового ключа |
| `action`   | Query        | string | —                | `register_start`       |

**Обмеження**: Максимум **5 Passkeys** на користувача. Якщо ліміт досягнуто — помилка.

**Відповідь (200):**

```json
{
  "success": true,
  "options": {
    "rp": { "id": "example.com", "name": "PayPaste" },
    "user": {
      "id": "<base64url>",
      "name": "<base64url>",
      "displayName": "PasskeyUser"
    },
    "challenge": "<base64url-challenge>",
    "pubKeyCredParams": [
      { "type": "public-key", "alg": -7 },
      { "type": "public-key", "alg": -257 }
    ],
    "timeout": 60000,
    "authenticatorSelection": {
      "authenticatorAttachment": null,
      "userVerification": "preferred",
      "residentKey": "preferred"
    },
    "attestation": "direct"
  }
}
```

**Алгоритми**: `-7` = ECDSA w/ SHA-256, `-257` = RSASSA-PKCS1-v1_5 w/ SHA-256.

#### `register_finish` — Завершення реєстрації

Приймає відповідь від автентифікатора, верифікує атестацію та зберігає публічний ключ.

**Заголовки:**

| Заголовок      | Обов'язковий | Опис               |
| -------------- | ------------ | ------------------ |
| `X-CSRF-Token` | ✅           | CSRF-токен для API |

**Тіло запиту (JSON):**

```json
{
  "credential": {
    "credentialId": "<base64url>",
    "clientDataJSON": "<base64url>",
    "attestationObject": "<base64url>",
    ...
  }
}
```

**Відповідь при успіху (200):**

```json
{
  "success": true,
  "redirect": "index.php"
}
```

**Поведінка для незареєстрованих користувачів:**

Якщо `$_SESSION['user_id']` не встановлено, автоматично створюється новий акаунт:

- Email: `passkey_<first_16_chars_of_credential_id>@paypaste.local`
- Пароль: випадковий (32 байти)
- Нікнейм: з сесії (`webauthn_nickname`)
- Баланс: 100 кредитів

#### `login_start` — Початок входу

Генерує челендж для аутентифікації. `allowCredentials` порожній — підтримується будь-який зареєстрований ключ (discoverable credentials).

**Відповідь (200):**

```json
{
  "success": true,
  "options": {
    "challenge": "<base64url-challenge>",
    "rpId": "example.com",
    "timeout": 60000,
    "userVerification": "preferred",
    "allowCredentials": []
  }
}
```

---

#### `login_finish` — Завершення входу

Верифікує assertion-підпис від автентифікатора та авторизує сесію.

**Тіло запиту (JSON):**

```json
{
  "credential": {
    "credentialId": "<base64url>",
    "clientDataJSON": "<base64url>",
    "authenticatorData": "<base64url>",
    "signature": "<base64url>",
    ...
  }
}
```

**Відповідь при успіху (200):**

```json
{
  "success": true,
  "redirect": "index.php"
}
```

**Алгоритм верифікації:**

1. Знайти Passkey за `credentialId` у БД
2. Верифікувати assertion-підпис за допомогою збереженого публічного ключа
3. Оновити лічильник підписів (`updateCounter()`) для захисту від replay
4. Авторизувати сесію через `AuthService::passkeyLogin()`

---

#### `confirm_delete_finish` — Підтвердження видалення акаунта

Вимагає володіння Passkey перед деструктивною операцією видалення акаунта.

**Тіло запиту:** як у `login_finish`.

**Відповідь при успіху (200):**

```json
{
  "success": true
}
```

Після успішної верифікації встановлюється `$_SESSION['passkey_confirmed_delete'] = true`, що дозволяє подальше видалення акаунта.

**Перевірки:**

- `$_SESSION['user_id']` обов'язковий
- Passkey має належати поточному користувачу (`passkey.user_id === session.user_id`)

---

#### `delete` — Видалення Passkey

Видаляє конкретний Passkey зі списку користувача.

**Параметри:**

| Поле           | Розташування | Тип    | Обов'язковий | Опис                     |
| -------------- | ------------ | ------ | ------------ | ------------------------ |
| `action`       | Query/POST   | string | ✅           | `delete`                 |
| `passkey_id`   | POST         | string | ✅           | ID Passkey для видалення |
| `X-CSRF-Token` | Header       | string | ✅           | CSRF-токен               |

**Відповідь при успіху (200):**

```json
{
  "success": true,
  "redirect": "settings.php"
}
```

**Перевірки:** Passkey має належати поточному користувачу.

## Webhooks

Webhook-ендпоінти приймають запити від зовнішніх сервісів (платіжних платформ) та обробляють колбеки.

### Donatello (`donatello.php`)

Обробляє сповіщення від платіжної платформи [Donatello](https://donatello.to) про надходження донатів (fiat-платежі в UAH).

| Параметр           | Значення                                         |
| ------------------ | ------------------------------------------------ |
| **URL**            | `/api/webhooks/donatello.php`                    |
| **Метод**          | `POST`                                           |
| **Автентифікація** | `X-Donatello-Token` / `X-Token` / `X-Key` header |

#### Верифікація запиту

Заголовок `X-Donatello-Token` (або `X-Token` / `X-Key`) порівнюється з `DONATELLO_TOKEN` із конфігурації. Якщо не співпадає → 401.

#### Формат вхідних даних

Donatello може надсилати дані у двох форматах:

**1. Одиничний донат (JSON):**

```json
{
  "message": "Поповнення order_abc123",
  "amount": 100,
  "pubId": "don_12345"
}
```

**2. Масив донатів (JSON):**

```json
{
  "content": [
    { "message": "Поповнення order_abc1", "amount": 25, "pubId": "don_111" },
    { "message": "Поповнення order_abc2", "amount": 250, "pubId": "don_222" }
  ]
}
```

**3. POST-форма:** Якщо JSON-парсинг не вдався, читається з `$_POST['message']`.

#### Логіка обробки

1. Парсинг `order_id` з поля `message` через regex: `/order_[a-zA-Z0-9]+/`
2. Пошук замовлення в БД: `Order::findById($order_id)`
3. Замовлення має бути в статусі `pending`
4. Розрахунок кредитів за тарифною сіткою:

| Сума (UAH) | Кредити             |
| ---------- | ------------------- |
| ≥ 250      | 1500                |
| ≥ 100      | 500                 |
| ≥ 25       | 100                 |
| < 25       | `floor(amount × 4)` |

5. Нарахування кредитів через `CreditService::credit()` (тип: `topup`)
6. Оновлення замовлення: `status = 'completed'`, `service = 'donatello'`

#### Відповідь

- **Успіх**: `{"success": true}` (200)
- **Не авторизовано**: `{"success": false, "message": "Не авторизовано"}` (401)

> 💡 **Ідемпотентність**: Якщо замовлення вже `completed` — обробка пропускається. `CreditService::credit()` використовує `idempotency_key` (`topup:donatello:<order_id>`) для захисту від дублювання.

### Telegram Stars (`telegram_stars.php`)

Webhook для обробки платежів через **Telegram Stars** (XTR-валюта). Цей скрипт працює як Telegram Bot Webhook — приймає `Update` об'єкти.

| Параметр           | Значення                                               |
| ------------------ | ------------------------------------------------------ |
| **URL**            | `/api/webhooks/telegram_stars.php`                     |
| **Метод**          | `POST`                                                 |
| **Автентифікація** | Неявна (Telegram надсилає на налаштований webhook URL) |

#### Flow обробки

Скрипт обробляє 4 типи Telegram Update:

**1. Команда `/start order_xxx` (message)**

Користувач відкриває бота з deep-link, що містить `order_id`. Бот відправляє повідомлення з inline-кнопками для вибору тарифу:

| Кнопка         | ⭐ Stars | Кредити |
| -------------- | -------- | ------- |
| 🥉 Базовий     | 50       | 100     |
| 🥈 Стандартний | 200      | 500     |
| 🥇 Преміум     | 500      | 1500    |

Callback-data формат: `tariff_<stars>_<order_id>` (напр. `tariff_50_order_abc123`)

**2. Натискання inline-кнопки (callback_query)**

Парсинг callback*data через regex: `/^tariff*([0-9]+)_(order_.+)$/`

Відправка Invoice через `sendInvoice`:

| Параметр Invoice   | Значення                     |
| ------------------ | ---------------------------- |
| `currency`         | `XTR`                        |
| `prices[0].amount` | кількість Stars (50/200/500) |
| `payload`          | `<order_id>\|<stars_amount>` |

**3. Pre-checkout query**

Обов'язковий обробник — підтверджує готовність прийняти оплату: `answerPreCheckoutQuery(ok: true)`.

**4. Успішна оплата (successful_payment)**

Парсинг `invoice_payload` → розбиття за `|` → отримання `order_id` та `stars`.

| Stars | Кредити |
| ----- | ------- |
| 50    | 100     |
| 200   | 500     |
| ≥ 500 | 1500    |

Нарахування через `CreditService::credit()` (тип: `topup`, service: `tg_stars`).

Оновлення замовлення: `status = 'completed'`, `service = 'tg_stars'`, `external_provider_id = telegram_payment_charge_id`.

Після успішного нарахування бот відправляє: `"✅ Оплата успішна! Вам нараховано X кредитів."`

#### Відповідь

Завжди повертається HTTP 200 (вимога Telegram API для уникнення повторної відправки).

> 💡 **Перевірка доступу**: Цей webhook не має явної верифікації підпису — безпека забезпечується тим, що URL webhook'а відомий лише Telegram API і не публікується.

### Підтвердження перегляду реклами (`verify_ad.php`)

AJAX-endpoint для підтвердження перегляду рекламних блоків (Adsterra Quest). Викликається з клієнтського JavaScript.

| Параметр           | Значення                      |
| ------------------ | ----------------------------- |
| **URL**            | `/api/webhooks/verify_ad.php` |
| **Метод**          | `POST`                        |
| **Автентифікація** | Сесія + CSRF                  |

#### Параметри запиту (POST-форма)

| Поле         | Тип    | Обов'язковий | Опис                                                                  |
| ------------ | ------ | ------------ | --------------------------------------------------------------------- |
| `csrf_token` | string | ✅           | CSRF-токен з сесії                                                    |
| `paste_id`   | string | ✅           | ID пасти, для якої виконується квест                                  |
| `ad_token`   | string | ✅           | Підписаний одноразовий токен (виданий `AdQuestService::issueToken()`) |

#### Логіка обробки (AdQuestService::verifyEvent)

**Модель «Ad Quest»**: Платна паста може бути розблокована безкоштовно після перегляду **3 рекламних подій**.

Алгоритм верифікації (14 кроків):

| #   | Перевірка                                               | Код причини відмови    |
| --- | ------------------------------------------------------- | ---------------------- |
| 1   | Rate Limit (12 подій/60с за користувачем, 300/хв за IP) | `rate_limited`         |
| 2   | Розшифрування та перевірка HMAC-підпису токена          | `bad_signature`        |
| 3   | Співпадіння `paste_id` у токені з переданим             | `paste_mismatch`       |
| 4   | Співпадіння `user_id` у токені з сесією                 | `user_mismatch`        |
| 5   | Співпадіння `identity_hash` (стабільний ідентифікатор)  | `identity_mismatch`    |
| 6   | Співпадіння `session_fp` (fingerprint браузера)         | `fingerprint_mismatch` |
| 7   | Термін дії токена (900с / 15 хв)                        | `expired`              |
| 8   | Співпадіння `quest_id` з поточним квестом               | `quest_mismatch`       |
| 9   | Наявність `nonce` у токені                              | `missing_nonce`        |
| 10  | Відсутність nonce у БД (захист від replay)              | `replay`               |
| 11  | Наявність nonce у сесії (виданий сервером)              | `unknown_nonce`        |
| 12  | Термін дії виданого nonce                               | `issued_expired`       |
| 13  | Мін. інтервал між подіями (10с)                         | `too_fast`             |
| 14  | Послідовність кроків                                    | `step_mismatch`        |

**Зарахування**: `INSERT IGNORE INTO ad_events` — гарантує ідемпотентність навіть при race condition.

#### Відповідь при успіху (200)

```json
{
  "success": true,
  "ads_watched": 2,
  "remaining": 1,
  "done": false,
  "next_token": "<base64url-signed-token>"
}
```

| Поле          | Тип          | Опис                                       |
| ------------- | ------------ | ------------------------------------------ |
| `ads_watched` | int          | Кількість зарахованих рекламних подій      |
| `remaining`   | int          | Залишилось подій до розблокування (0–3)    |
| `done`        | bool         | Чи завершено квест (≥3 подій)              |
| `next_token`  | string\|null | Токен для наступної події (null якщо done) |

#### Відповідь при завершенні квесту

```json
{
  "success": true,
  "ads_watched": 3,
  "remaining": 0,
  "done": true,
  "next_token": null
}
```

#### Відповідь при відмові (400 / 429)

```json
{
  "success": false,
  "message": "Опис помилки",
  "reason": "replay",
  "ads_watched": 1,
  "remaining": 2,
  "done": false
}
```

#### Коди помилок

| HTTP | Умова                                      |
| ---- | ------------------------------------------ |
| 405  | Метод не POST                              |
| 403  | Невірний CSRF-токен                        |
| 400  | Відсутній `paste_id` або `ad_token`        |
| 400  | Будь-яка з 14 перевірок токена не пройдена |
| 429  | Rate limit (або `reason: rate_limited`)    |

> 🔒 **Безпека**: Токен підписується HMAC-SHA256 з використанням `COOKIE_SECRET`. Nonce — одноразовий, записується у БД з UNIQUE-обмеженням. Сесія використовується лише як кеш, авторитетним джерелом правди є БД (`ad_events`).

## Коди помилок

Зведена таблиця HTTP-кодів, що використовуються в API:

| Код | Значення           | Де використовується                                                           |
| --- | ------------------ | ----------------------------------------------------------------------------- |
| 200 | OK                 | Успішна відповідь (GET, DELETE, webhooks)                                     |
| 201 | Created            | Паста успішно створена (POST pastes.php)                                      |
| 400 | Bad Request        | Валідація: відсутні обов'язкові параметри, невірний формат, помилка модерації |
| 401 | Unauthorized       | Невірний JWT / API-ключ / webhook-токен / OAuth-помилка                       |
| 402 | Payment Required   | Недостатньо кредитів (api_usage, creation_fee, purchase)                      |
| 403 | Forbidden          | Доступ заборонено (приватна паста, CSRF-помилка, невладнаний файл)            |
| 404 | Not Found          | Паста/замовлення/файл не знайдено                                             |
| 405 | Method Not Allowed | Непідтримуваний HTTP-метод                                                    |
| 429 | Too Many Requests  | Rate limit перевищено                                                         |
| 500 | Internal Error     | (Внутрішня помилка, не очікується у відповідях)                               |

### Потік даних: Рекламний квест (Ad Quest)

```
Клієнт (app.js)
  │
  ├── GET /view.php?id=p_paid123  →  шаблон показує блок реклами
  │
  ├── AdQuestService::issueToken()  ←  сервер генерує підписаний токен
  │     (вбудовано у HTML через data-атрибут або AJAX)
  │
  ├── Користувач переглядає рекламу (Adsterra)
  │
  ├── POST /api/webhooks/verify_ad.php
  │     csrf_token + paste_id + ad_token
  │     → AdQuestService::verifyEvent()
  │     ← { success: true, ads_watched: 1, remaining: 2, next_token: "..." }
  │
  ├── ... повторити 3 рази ...
  │
  └── Після 3-ї події:
       ← { done: true, next_token: null }
       → Клієнт розблоковує контент пасти
```

### Потік даних: REST API

```
Зовнішній клієнт (curl / програма)
  │
  ├── POST /api/auth_token.php  { api_key: "ak_xxx" }
  │     ← { access_token: "eyJ...", token_type: "Bearer", expires_in: 604800 }
  │
  ├── POST /api/pastes.php  Authorization: Bearer eyJ...
  │     { title: "...", content: "...", is_paid: true, view_cost: 50 }
  │     → authenticate_api() → списання 1 кредиту (api_usage)
  │     → PasteService::create() → списання creation_fee
  │     → Moderation::localCheck() + Queue::push(TYPE_MODERATION_CHECK)
  │     ← 201 { id: "p_xxx", moderation_status: "pending" }
  │
  ├── GET /api/pastes.php?id=p_xxx  Authorization: Bearer eyJ...
  │     → authenticate_api() → списання 1 кредиту
  │     → PasteService::unlock() → списання view_cost (якщо платна)
  │     ← 200 { id, title, content, ... }
  │
  └── DELETE /api/pastes.php?id=p_xxx  Authorization: Bearer eyJ...
        → authenticate_api() → списання 1 кредиту
        → PasteService::delete()
        ← 200 { success: true }
```
