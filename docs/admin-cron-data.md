## 1. admin/ — Панель адміністратора

### 1.1 Загальний огляд

Папка `admin/` містить повноцінну панель управління проектом PayPaste. Доступна лише користувачам з роллю `admin`. Забезпечує повний CRUD над пастами, користувачами та перегляд фінансових транзакцій і стану черги фонових задач.

**Файли:**

| Файл               | Призначення                                 |
| ------------------ | ------------------------------------------- |
| `check_admin.php`  | Мідлвер перевірки прав адміністратора       |
| `index.php`        | Головний дашборд зі статистикою             |
| `pastes.php`       | Список паст з пошуком та пагінацією         |
| `edit_paste.php`   | Форма редагування пасти                     |
| `delete_paste.php` | Обробник видалення пасти (POST)             |
| `users.php`        | Список користувачів з пошуком та пагінацією |
| `delete_user.php`  | Обробник видалення користувача (POST)       |
| `transactions.php` | Історія транзакцій з фільтрами              |
| `queue.php`        | Моніторинг черги фонових задач              |

### 1.2 Автентифікація та авторизація

**Файл:** `check_admin.php`

Мідлвер, який підключається на початку кожного адмін-файлу. Механізм:

1. Запускає сесію (`session_start()`)
2. Перевіряє наявність `$_SESSION['user_id']` та `$_SESSION['role']`
3. Якщо роль !== `'admin'` — редирект на `../index.php` (головна сайту)
4. Інакше — виконання продовжується

```php
// Логіка check_admin.php
session_start();
if (!isset($_SESSION['user_id']) || !isset($_SESSION['role']) || $_SESSION['role'] !== 'admin') {
    header("Location: ../index.php");
    exit();
}
```

> ⚠️ **Важливо:** Усі адмін-сторінки починаються з `include_once __DIR__ . "/check_admin.php"`, що гарантує захист навіть при прямому зверненні до URL.

### 1.3 index.php — Дашборд статистики

Головна сторінка адмін-панелі. Відображає:

**Загальна статистика (4 картки):**

| Метрика               | Джерело                    | Опис                          |
| --------------------- | -------------------------- | ----------------------------- |
| 📝 Всього Паст        | `Paste::countAll()`        | Загальна кількість паст у БД  |
| 👥 Користувачів       | `User::countAll()`         | Зареєстрованих користувачів   |
| 💵 Кредитів поповнено | `Transaction::sumTopups()` | Сума усіх поповнень кредитів  |
| 📊 Транзакцій         | `Transaction::count()`     | Загальна кількість транзакцій |

**Метрики черги задач (4 картки):**

| Метрика     | Опис                                      |
| ----------- | ----------------------------------------- |
| 📬 У черзі  | Кількість задач у статусі `queued`        |
| 💀 Мертві   | Задачі, що вичерпали ліміт спроб (`dead`) |
| 🔄 Ретраї   | Загальна кількість повторних спроб        |
| ⏱️ Сер. час | Середній час виконання задач (секунди)    |

Дані черги отримуються через `Queue::getMetrics()` (див. [Система черги](#25-система-черги-queue)).

Якщо є помилки — відображається таблиця «Останні помилки» з 5 найсвіжіших failed/dead задач.

### 1.4 pastes.php — Управління пастами

Список усіх паст з функціями перегляду, редагування та видалення.

**Можливості:**

- 🔍 **Пошук** за заголовком або вмістом (GET-параметр `search`)
- 📄 **Пагінація** — 25 записів на сторінку
- 👁️ **Перегляд** — посилання відкриває пасту на сайті (`view.php`)
- ✏️ **Редагування** — перехід на `edit_paste.php`
- 🗑️ **Видалення** — форма POST з CSRF на `delete_paste.php`

**Відображувані поля:**

| Стовпець      | Опис                                                  |
| ------------- | ----------------------------------------------------- |
| ID            | Перші 10 символів ID пасти                            |
| Назва         | Заголовок пасти                                       |
| Автор         | Email автора або «Anon/Guest»                         |
| Дата          | Дата створення                                        |
| Доступ / Ціна | Лейбли: 💰 платна (з ціною), безкоштовна, 🔒 приватна |

Дані: `Paste::getAllPastes($perPage, $offset, $search)`, автор визначається через `User::findById()`.

### 1.5 edit_paste.php — Редагування пасти

Форма редагування існуючої пасти. Доступна через GET-параметр `id`.

**Редаговані поля:**

- Назва / Тема (`title`)
- Текст контенту (`content`, textarea, моноширинний шрифт)
- Приватність (`is_private`, чекбокс)
- Платність (`is_paid`, чекбокс з попередженням) + ціна (`view_cost`)
- Дата закінчення (`expires_at`, datetime-local, порожнє = вічно)

**Обробка POST:**

1. Перевірка CSRF через `verify_csrf()`
2. Оновлення властивостей об'єкта `$paste`
3. Валідація: контент не порожній
4. Збереження через `$paste->update()`

> 💡 При встановленні «Платна» динамічно з'являється поле «Ціна за перегляд» (JavaScript-логіка).

### 1.6 delete_paste.php — Видалення пасти

Обробник POST-запиту на видалення. Логіка:

1. Перевірка CSRF через `verify_csrf()`
2. Отримання ID пасти з `$_POST['id']`
3. Виклик `PasteService::delete($paste_id, null)` — `null` означає, що це адмін
4. Редирект на `pastes.php`

`PasteService::delete()` також видаляє прикріплені файли з `data/uploads/` (див. [uploads/](#33-uploads--файли-вкладень-паст)).

### 1.7 users.php — Управління користувачами

Список користувачів з пошуком та можливістю видалення.

**Можливості:**

- 🔍 Пошук за Email або Нікнеймом
- 📄 Пагінація — 25 на сторінку
- 🗑️ Видалення не-адмін користувачів (кнопка недоступна для адміністраторів)

**Відображувані поля:**

| Стовпець      | Опис                                        |
| ------------- | ------------------------------------------- |
| Користувач    | Нікнейм + Email                             |
| Баланс        | Кредити                                     |
| Роль          | Адмін (червоний лейбл) / Користувач (синій) |
| OAuth         | GitHub / Telegram лейбли, якщо підключені   |
| Зареєстровано | Дата реєстрації                             |

Дані: `User::getAll($perPage, $offset, $search)`.

> ⚠️ **Захист:** Кнопка видалення заблокована для користувачів з роллю `admin`.

### 1.8 delete_user.php — Видалення користувача

Обробник POST-запиту на видалення користувача:

1. Перевірка CSRF через `verify_csrf()`
2. Отримання ID з `$_POST['id']`
3. Делегація до `AuthService::deleteByAdmin($user_id)`
4. Редирект на `users.php`

## 2. cron/ — Фонові задачі та Worker

### 2.1 Загальний огляд

Папка `cron/` містить скрипти для асинхронної обробки задач, які не повинні блокувати HTTP-запити користувачів. Це серверна частина системи черги повідомлень.

**Файли:**

| Файл            | Призначення                                                                     |
| --------------- | ------------------------------------------------------------------------------- |
| `worker.php`    | Уніфікований Worker — основний процесор черги                                   |
| `ai_worker.php` | Зворотно-сумісний wrapper → делегує `worker.php` з `--type=moderation_rewrite` |
| `cleanup.php`   | Очищення протермінованих паст та старих задач                                   |

### 2.2 worker.php — Уніфікований Worker

Основний скрипт для обробки задач з черги (`jobs` таблиця). Замінює колишній `ai_worker.php`, забезпечує retry, timeout, логування та контрольовану деградацію.

**Режими запуску:**

```bash
# Одноразовий батч (обробити наявні задачі і вийти)
php cron/worker.php

# Безперервний режим (для VPS — працює постійно)
php cron/worker.php --daemon

# Лише задачі конкретного типу
php cron/worker.php --type=email_verify
```

**Конфігурація:**

| Параметр         | Значення | Опис                                  |
| ---------------- | -------- | ------------------------------------- |
| `BATCH_SIZE`     | 10       | Кількість задач за один батч          |
| `SLEEP_BETWEEN`  | 2 сек    | Затримка між батчами в daemon-режимі  |
| `CURL_TIMEOUT`   | 30 сек   | Таймаут для зовнішніх API             |
| `set_time_limit` | 300 сек  | Ліміт виконання PHP на батч           |
| `NO_SESSION`     | `true`   | Worker не ініціалізує сесії/CSRF/кукі |

**Обробники задач:**

#### handleModerationCheck($payload)

Перевірка контенту через **OpenAI Moderation API**.

1. Отримує `paste_id` та `content` з payload
2. Перевіряє, що паста ще існує та має статус `pending`
3. Викликає `Moderation::checkExternal($content)`
4. Якщо порушень немає → статус `approved`
5. Якщо є порушення → статус `rejected`, причини зберігаються у `moderation_result`

> Див. деталі модерації у [includes/Moderation.php → docs/includes.md](includes.md)

#### handleModerationRewrite($payload)

AI-перефразування контенту через **Ollama**.

1. Отримує `paste_id` з payload
2. Перевіряє, що паста має `is_pending_rewrite = 1`
3. Викликає `Moderation::rewrite($paste['content'])` — перефразування через Ollama
4. Оновлює контент пасти, знімає прапорець `is_pending_rewrite`, встановлює `moderation_status = 'approved'`
5. Синхронізує теги через `$pasteObj->syncTags()`

#### handleEmailVerify($payload)

Відправка верифікаційного листа.

1. Отримує `to` (email) та `code` (OTP)
2. Підставляє код у шаблон `templates/email_verify.html`
3. Відправляє через `Mailer::sendDirect()`

#### handleEmailChanged($payload)

Повідомлення про зміну email-адреси.

1. Отримує `old_email` та `new_email`
2. Підставляє у шаблон `templates/email_changed.html`
3. Відправляє на стару адресу через `Mailer::sendDirect()`

#### handleEmailCustom($payload)

Довільне email-повідомлення.

1. Отримує `to`, `subject`, `html`
2. Відправляє напряму через `Mailer::sendDirect()`

**Головний цикл:**

```
┌───────────────────────────────┐
│ Worker запущено (single/daemon)│
└──────────┬────────────────────┘
           │
           ▼
┌──────────────────────┐
│ Queue::pop(BATCH_SIZE)│◄───────┐
└──────────┬───────────┘        │
           │                    │
     ┌─────┴─────┐              │
     │ Порожньо?  │              │
     └─────┬─────┘              │
      Так  │  Ні                │
     ┌─────┴─────┐  ┌──────────┴─────────┐
     │ Daemon?   │  │ Обробити кожну job  │
     │→ sleep+↺  │  │ handler(payload)    │
     │→ exit     │  │ → Queue::complete() │
     └───────────┘  │ або Queue::fail()   │
                    └──────────┬──────────┘
                               │
                    Daemon? ──├── Так → sleep + ↺
                               └── Ні  → exit
```

**Логування:** Усі події пишуться у `data/logs/worker.log` через `workerLog()`:

```
[2026-05-11 14:30:00] === Worker запущено (single) ===
[2026-05-11 14:30:00] Обробка moderation_check [j_a1b2c3d4] (спроба 1)
[2026-05-11 14:30:02] Завершено moderation_check [j_a1b2c3d4]
[2026-05-11 14:30:02] Worker зупинено (оброблено ітерацій: 1)
```

### 2.3 ai_worker.php — Wrapper для AI-переписування

Зворотно-сумісний wrapper, що делегує роботу уніфікованому `worker.php`:

```php
// Додаємо --type=moderation_rewrite до аргументів
$argv[] = '--type=moderation_rewrite';
$argc = count($argv);
require_once __DIR__ . '/worker.php';
```

Запуск: `php cron/ai_worker.php` → еквівалентно `php cron/worker.php --type=moderation_rewrite`

> 📝 Збережено для зворотної сумісності. Рекомендується використовувати `worker.php` напряму.

### 2.4 cleanup.php — Очищення протермінованих даних

Скрипт для регулярного видалення протермінованих паст та старих задач черги.

**Запуск:**

```bash
# Рекомендовано: через CLI
php cron/cleanup.php

# Альтернатива: через HTTP з параметром ?force=1
# (не рекомендується для продакшену)
```

Захист від випадкового HTTP-виклику: скрипт перевіряє `php_sapi_name() !== 'cli'` і вимагає `?force=1` для веб-доступу.

**Логіка:**

1. **Очищення паст** — `PasteService::cleanupExpired()`:
   - SELECT усіх паст, де `expires_at IS NOT NULL AND expires_at <= NOW()`
   - Послідовне видалення кожної пасти через `PasteService::delete()` (включаючи файли з `data/uploads/`)
   - Повертає кількість видалених паст

2. **Очищення черги** — `Queue::cleanup()`:
   - DELETE задач зі статусами `completed` або `dead`, старіших за 7 днів
   - Повертає кількість видалених задач

**Вивід:**

```
[2026-05-11 14:00:00] Початок очищення протермінованих паст...
[2026-05-11 14:00:00] Очищення завершено. Видалено паст: 3
[2026-05-11 14:00:00] Очищення черги: видалено 12 старих задач.
```

### 2.5 Система черги (Queue)

Клас `Queue` (у `includes/Queue.php`) — це єдиний механізм постановки та управління фоновими задачами. Використовує MySQL як backend (таблиця `jobs`).

**Типи задач:**

| Константа                 | Значення             | Опис                         | Обробник у worker.php     |
| ------------------------- | -------------------- | ---------------------------- | ------------------------- |
| `TYPE_MODERATION_CHECK`   | `moderation_check`   | Перевірка OpenAI Moderation  | `handleModerationCheck`   |
| `TYPE_MODERATION_REWRITE` | `moderation_rewrite` | Перефразування Ollama        | `handleModerationRewrite` |
| `TYPE_EMAIL_VERIFY`       | `email_verify`       | Верифікаційний лист          | `handleEmailVerify`       |
| `TYPE_EMAIL_CHANGED`      | `email_changed`      | Повідомлення про зміну email | `handleEmailChanged`      |
| `TYPE_EMAIL_CUSTOM`       | `email_custom`       | Довільне повідомлення        | `handleEmailCustom`       |

**Життєвий цикл задачі:**

```
push() → [queued] → pop() → [processing] → complete() → [completed]
                                          ↘ fail() → [queued] (retry з backoff)
                                                      або → [dead] (вичерпано спроби)
```

**Ключові методи:**

| Метод                                                                        | Опис                                                                                                  |
| ---------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------- |
| `Queue::push($type, $payload, $idempotencyKey, $maxAttempts, $delaySeconds)` | Постановка задачі. Генерує ID `j_` + 16 hex символів. Ідемпотентність за ключем запобігає дублюванню. |
| `Queue::pop($limit, $type)`                                                  | Отримання задач для обробки. SELECT + FOR UPDATE + UPDATE (claim). Звільняє завислі задачі.           |
| `Queue::complete($jobId)`                                                    | Позначення задачі завершеною                                                                          |
| `Queue::fail($jobId, $error)`                                                | Retry з експоненційним backoff (30×2^n сек) або перевод у `dead` при вичерпанні спроб                 |
| `Queue::getStatus($jobId)`                                                   | Статус конкретної задачі                                                                              |
| `Queue::getMetrics()`                                                        | Агреговані метрики для адмін-панелі                                                                   |
| `Queue::cleanup($olderThanDays)`                                             | Видалення старих completed/dead задач                                                                 |
| `Queue::releaseStuck()`                                                      | Звільнення завислих processing-задач (таймаут 300 сек)                                                |

**Конфігурація черги:**

| Параметр                  | Значення | Опис                                                   |
| ------------------------- | -------- | ------------------------------------------------------ |
| `DEFAULT_MAX_ATTEMPTS`    | 3        | Максимальна кількість спроб обробки                    |
| `BACKOFF_BASE_SECONDS`    | 30       | Базова затримка ретраю (30, 60, 120 сек)               |
| `CLAIM_TIMEOUT_SECONDS`   | 300      | Час, після якого processing-задача вважається завислою |
| `CLEANUP_OLDER_THAN_DAYS` | 7        | Дні для очищення старих задач                          |

**Fallback при dead-задачах (контрольована деградація):**

| Тип задачі           | Fallback-поведінка                                                           |
| -------------------- | ---------------------------------------------------------------------------- |
| `moderation_check`   | Паста auto-approved (локальна перевірка вже пройшла синхронно при створенні) |
| `moderation_rewrite` | Прапорець `is_pending_rewrite` знімається, оригінальний текст залишається    |
| `email_*`            | Логування помилки, адмін бачить у панелі                                     |

> 📝 Lazy-механізми (ліниве видалення протермінованих паст при доступі через `Paste::findById()`, fallback при недоступності worker) є навмисними демонстраційними функціями. Не видаляти їх лише через наявність фонового worker.

### 2.6 Налаштування Cron

Рекомендовані записи crontab:

```bash
# Worker — кожну хвилину (одноразовий батч)
* * * * * php /path/to/cron/worker.php >> /path/to/data/logs/worker.log 2>&1

# Або: daemon-режим (тільки для VPS, не для crontab)
# php /path/to/cron/worker.php --daemon

# Очищення протермінованих паст — кожну годину
0 * * * * php /path/to/cron/cleanup.php >> /path/to/data/logs/worker.log 2>&1
```

> Див. деталі розгортання у [architecture.md](architecture.md) (розділ «Розгортання»)

## 3. data/ — Файлове сховище

### 3.1 Загальний огляд та структура

Папка `data/` — це захищене файлове сховище проекту. Містить завантажені користувачами файли та логи worker-а.

```
data/
├── .htaccess          # Заборона прямого доступу через веб
├── uploads/           # Файли вкладень паст
│   ├── p_69e8a947a9fb4.png
│   ├── p_69e8aae28a0f2.png
│   └── ...
└── logs/              # Логи фонового Worker-а
    ├── .gitkeep
    └── worker.log     # Створюється автоматично worker.php
```

### 3.2 .htaccess — Захист прямого доступу

Файл `data/.htaccess` блокує будь-який прямий веб-доступ до вмісту папки `data/`:

```apache
# Заборона будь-якого прямого доступу до даних
<IfModule mod_authz_core.c>
    Require all denied
</IfModule>
<IfModule !mod_authz_core.c>
    Order Deny,Allow
    Deny from all
</IfModule>
```

Це критичний захисний механізм:

- ✅ Блокує прямі посилання типу `/data/uploads/secret_file.pdf`
- ✅ Працює на Apache 2.4 (`mod_authz_core`) та Apache 2.2 (fallback)
- ✅ Файли доступні лише через проксі `api/download.php`

> ⚠️ **НЕ ВИДАЛЯТИ** цей файл! Він захищає вкладення від несанкціонованого доступу.

### 3.3 uploads/ — Файли вкладень паст

Папка `data/uploads/` містить файли, завантажені користувачами як вкладення до паст.

**Іменування файлів:**

Формат: `{paste_id}.{extension}`

Приклади:

- `69e8a947a9fb4.png` — PNG-зображення для пасти з ID `69e8a947a9fb4`
- `a1b2c3d4e5f6.zip` — ZIP-архів для пасти з ID `a1b2c3d4e5f6`

**Обробка завантаження** (у `PasteService::handleFileUpload()`):

| Крок                 | Опис                                                       |
| -------------------- | ---------------------------------------------------------- |
| Перевірка розміру    | Максимум 5 МБ                                              |
| Створення директорії | `mkdir($uploadDir, 0755, true)` якщо не існує              |
| Витяг розширення     | `pathinfo($fileName, PATHINFO_EXTENSION)` — нижній регістр |
| Перевірка MIME       | `mime_content_type()` + відповідність розширенню           |
| Збереження           | `move_uploaded_file()` у `data/uploads/{paste_id}.{ext}`   |

**Дозволені типи файлів:**

| Розширення    | MIME-тип                                          | Опис            |
| ------------- | ------------------------------------------------- | --------------- |
| `jpg`, `jpeg` | `image/jpeg`                                      | JPEG-зображення |
| `png`         | `image/png`                                       | PNG-зображення  |
| `gif`         | `image/gif`                                       | GIF-зображення  |
| `zip`         | `application/zip`, `application/x-zip-compressed` | ZIP-архів       |
| `pdf`         | `application/pdf`                                 | PDF-документ    |
| `txt`         | `text/plain`                                      | Текстовий файл  |

> Безпека: перевірка і розширення, і MIME-типу одночасно запобігає обхідним атакам (напр., `.jpg` з PHP-кодом всередині).

**Видалення файлів:**

При видаленні пасти (`PasteService::delete()`) або очищенні протермінованих (`PasteService::cleanupExpired()`), файли видаляються з диска:

```php
$files = glob($uploadDir . $pasteId . '.*');
foreach ($files as $file) {
    if (is_file($file)) unlink($file);
}
```

### 3.4 logs/ — Логи Worker-а

Папка `data/logs/` містить лог-файли фонового worker-а.

| Файл         | Опис                                            |
| ------------ | ----------------------------------------------- |
| `worker.log` | Основний лог Worker-а (створюється автоматично) |
| `.gitkeep`   | Зберігає порожню папку в Git                    |

**Формат записів worker.log:**

```
[YYYY-MM-DD HH:MM:SS] Повідомлення
```

**Приклади записів:**

```
[2026-05-11 14:30:00] === Worker запущено (single) ===
[2026-05-11 14:30:00] Обробка moderation_check [j_a1b2c3d4] (спроба 1)
[2026-05-11 14:30:01] moderation_check: паста 69e8a947a9fb4 APPROVED (OpenAI чисто)
[2026-05-11 14:30:01] Завершено moderation_check [j_a1b2c3d4]
[2026-05-11 14:30:02] Worker зупинено (оброблено ітерацій: 1)
```

Логування здійснюється через функцію `workerLog()` у `cron/worker.php` з `FILE_APPEND | LOCK_EX` для безпечного паралельного запису.

### 3.5 Проксі завантажень через API

Оскільки прямий доступ до `data/uploads/` заблоковано `.htaccess`, файли видаються через API-проксі:

**Ендпоінт:** `api/download.php`

Логіка:

1. Перевірка прав доступу (автор, покупець або адмін)
2. Читання файлу з `data/uploads/`
3. Відправка з відповідними заголовками (`Content-Type`, `Content-Disposition`)

> Див. деталі у [docs/api.md](api.md)

| Скрипт        | Залежності                                                                                                            |

| Компонент              | Взаємодія                                                                    |
